Discussion: 

·             Compare the run-times of the six algorithms and tell me if they came out the way you expected them to. 

·             Compare the run-times for the different input sizes N for each of the sorting algorithms and tell me if they came out the way you expected them to. 

·             Do the empirical runtimes show that the theoretical big-O runtimes are really correct? 

o            For example, if you have a O(n2) algorithm, you would expect that as the input size N doubles, the empirical runtimes should go up approximately by a factor of 4 because if T(n) is O(n2) then T(2*n) is O((2*n)2). 

·             What are the constants for your algorithms? 

o            Remember if the empirical runtime T(n) is O(n2) then T(n) <= c * n2, and solve for c.

·             Overall, which version has the best empirical run time?

        NOTE: The word "empirical" means "observed" or "experimental"


Programming Help:

Tip 1:

If you are using Java, you can generate a random integer between UPPER and LOWER by using the command:

int n = (int)(LOWER + Math.random() * (UPPER-LOWER+1));

OR

import java.util.Random;

Random gen = new Random();

int n = gen.nextInt(UPPER) + LOWER; 

Tip 2:

After you make sure your sorting algorithms work, you can time them in milliseconds by getting the current time before and after the operation you want to time:

long start, end, runtime;
start = System.currentTimeMillis();
// Call your sorting function here (but initialize array BEFORE timing!) ...
end = System.currentTimeMillis();
runtime = end - start; // in milliseconds
